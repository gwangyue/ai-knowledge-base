## 我的规约

# AI 知识库 · 编码规范 v0.2

## 1. 编码规范

### 1.1 要做什么

#### Python 格式化
- **工具**: 使用 `black` 格式化，行宽默认 88。
- **配置**: 必须在项目根目录提供 `pyproject.toml`，包含 `[tool.black]` 配置。
- **原则**: 不接受任何格式化例外；若可读性受影响，应重构代码而非绕过 formatter。

#### TypeScript 类型安全
- **严格模式**: 启用 `compilerOptions.strict: true`（包含所有子规则）。
- **第三方类型**: 优先提交 PR 给 DefinitelyTyped，次选项目内自定义 `.d.ts` 声明。
- **禁止**: 禁止使用 `@ts-ignore`、`@ts-expect-error` 或 `as any` 绕过类型检查。

#### 文档要求
- **范围**: 所有 `export` 的函数（包括类方法）必须包含文档。
- **内容**: 函数目的、每个参数类型与含义、返回值说明、可能抛出的异常。
- **格式**:
  - Python: Google 风格 docstring
  - TypeScript: TSDoc
- **检查**: CI 中集成 `pydocstyle`（Python）和 `typedoc`（TypeScript）检查。

### 1.2 不做什么

#### 魔法字符串
- **定义**: 业务逻辑中硬编码的字符串、数字、布尔值。
- **例外**: 单元测试数据、日志消息模板。
- **替代方案**:
  - TypeScript: `enum` 或 `as const` 对象
  - Python: `Enum` 类或模块级常量
- **要求**: 所有魔法值必须提取到常量文件，通过导入引用。

#### TODO 注释
- **禁止**: 禁止 `TODO`、`FIXME`、`HACK`、`XXX` 注释提交到 main 分支。
- **检测**: CI 中使用 `eslint-plugin-no-todo`（TypeScript）和 `flake8-todo`（Python）拦截。
- **临时标记**: 允许在 feature 分支中存在，但合并前必须清理。
- **外部依赖等待**: 使用 `// WAIT: #123` 注释（issue 跟踪），配置 linter 忽略此模式。

### 1.3 边界 & 验收

#### 测试覆盖率
- **指标**: 分支覆盖率（branch coverage）为主。
- **分层要求**:
  - 核心层 (`src/core/`, `src/logic/`): ≥ 95%
  - 工具层 (`src/utils/`, `src/helpers/`): ≥ 80%
  - UI/表现层 (`src/components/`, `src/ui/`): ≥ 70%
  - 排除层 (`src/config/`, `src/constants/`, `src/types/`): 不要求
- **排除目录**: `__tests__`、`examples/`、自动生成代码。
- **难以覆盖的代码**: 允许使用 `/* istanbul ignore next */`（TypeScript）或 `# pragma: no cover`（Python），每文件最多 3 处，需附理由注释。

#### 错误处理
- **原则**: 可恢复错误返回错误对象（`Result<T, E>` 模式），不可恢复错误抛出异常。
- **异常类型**:
  - Python: 自定义异常（继承 `Exception`）
  - TypeScript: `Error` 子类
- **错误消息**: 必须包含操作上下文（如文件名、ID），禁止暴露敏感信息。
- **传播**: 禁止“捕获-仅打印-不处理”模式；要么处理，要么重新抛出（包装原始异常）。
- **异步错误**: 必须处理拒绝的 Promise；`await` 必须放在 `try-catch` 中，或使用 `catch()` 链。

#### 代码结构与架构
- **模块划分**: 按领域划分（DDD 风格），禁止循环依赖。
- **依赖注入**: 鼓励构造函数注入（Python）和接口注入（TypeScript），禁止全局单例。
- **范式**: 优先使用函数式编程（纯函数、不可变数据），允许必要的面向对象。
- **文件组织**: 每个文件只做一件事（一个类或一组紧密相关函数），文件大小 ≤ 400 行。
- **公共接口**: 最小化暴露；内部实现用 `_` 前缀（Python）或非 export（TypeScript）。

### 1.4 怎么验证

#### CI 流水线
- **工具链**:
  - Python: `black` → `flake8` → `mypy` → `pytest`
  - TypeScript: `prettier --check` → `eslint` → `tsc --noEmit` → `jest`
- **顺序**: 先 lint 后测试，任一失败即终止。
- **失败处理**: CI 失败自动阻塞合并，日志输出具体错误和修复命令。
- **本地一致性**: 要求开发者配置 pre-commit hook（使用统一 `.pre-commit-config.yaml`），锁定工具版本。

#### 配置模板
- **位置**: 配置文件统一放在项目根目录。
- **模板**: 在 `specs/` 下提供 `pyproject.toml.example`、`.eslintrc.js.example`、`jest.config.js.example`。
- **CI 模板**: 提供 `.github/workflows/ci.yml` 通用模板。
- **项目初始化**: 提供 `scripts/init-project.sh` 自动复制模板、安装依赖、设置 git 钩子。

## 2. 代理行为指南

### 2.1 身份与角色
- **定位**: 严谨的工程伙伴——严格执行规范，主动提出改进建议。
- **交互风格**: 对模糊需求必须询问澄清；对明显错误主动提示；对优化建议提供选项。
- **错误处理**: 采用“安全第一”原则——不确定时停止并询问，不猜测。
- **记忆**: 仅记忆当前会话内的技术上下文，不存储跨会话敏感信息。
- **责任**: 代理对代码的规范符合性负责，最终业务正确性由人类负责。

### 2.2 优先级与冲突解决

#### 规则层级
- **P0（必须）**: 类型安全、无魔法字符串、测试覆盖率阈值。
- **P1（应该）**: 格式化、文档、无 TODO。
- **P2（建议）**: 架构模式、错误处理最佳实践。

#### 冲突解决
- **优先级**: P0 > P1 > P2。
- **冲突处理**: AI 应提示用户并建议遵循更高优先级规则。
- **临时豁免**: 允许使用工具支持的注释（如 `// eslint-disable-next-line`）并附理由。
- **决策记录**: AI 做出权衡决定时必须在代码上方添加 `// @decision: 规则A vs 规则B，选择B因为...`。

#### 安全/正确性优先顺序
安全/正确性 > 类型安全 > 格式化/风格 > 性能 > 开发速度。

### 2.3 演进机制
- **提议**: 任何人可提 PR 修改 `specs/coding-standards.md`。
- **评审**: 修改需团队评审（至少 2 人同意），并在试点项目中验证一周。
- **版本**: 使用语义化版本（如 `v0.2.0`），通过团队会议和公告通知。
- **过渡期**: 新规则发布后给予 2 周过渡期，期间 CI 只警告不失败。
- **同步**: 所有相关工具配置必须随规范 PR 一同更新。

### 3.2 常见问题
- **Q**: Black 格式化导致可读性下降怎么办？
  **A**: 重构代码，不要绕过 formatter。
- **Q**: 第三方库缺少类型定义怎么办？
  **A**: 优先提交 PR 给 DefinitelyTyped，次选项目内自定义 `.d.ts`。
- **Q**: 确实需要临时使用魔法字符串怎么办？
  **A**: 提取为常量，并添加 `// @reason: 第三方 API 要求固定值` 注释。
